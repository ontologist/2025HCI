### Slide 2: What is Wireframing? | ワイヤーフレーミングとは？

#### Wireframing Fundamentals (3 minutes) | ワイヤーフレーミングの基礎（3分）

**English Speaking Points:**
- "Wireframing is like creating a blueprint for a building - it shows the structure without the decorative details."
- "Think of wireframes as the skeleton of your interface. They show where content goes, how elements are organized, and the overall layout."
- "Wireframes are intentionally simple - grayscale, basic shapes, placeholder text. This helps everyone focus on structure, not visual design."
- "The goal is to explore many layout ideas quickly. You can create 5 different wireframes in the time it takes to do one detailed design."

**Japanese Speaking Points:**
- "ワイヤーフレーミングは建物の設計図を作成するようなものです - 装飾的な詳細なしで構造を示します。"
- "ワイヤーフレームをインターフェースの骨格と考えてください。コンテンツがどこに行くか、要素がどのように整理されているか、全体的なレイアウトを示します。"
- "ワイヤーフレームは意図的にシンプルです - グレースケール、基本的な形、プレースホルダーテキスト。これにより、全員が視覚デザインではなく構造に焦点を当てることができます。"
- "目標は多くのレイアウトアイデアを迅速に探索することです。1つの詳細なデザインを行う時間で5つの異なるワイヤーフレームを作成できます。"

**Real-World Examples:**
- Show examples of wireframes vs. final designs
- Point out how wireframes reveal structure
- Demonstrate speed of wireframe iteration
- Compare to architectural blueprints

**Interactive Element:**
- Ask: "Who has sketched a layout on paper before building something?"
- Discuss: "Why do we sketch before committing to final designs?"

### Slide 3: What is Prototyping? | プロトタイピングとは？

#### Prototyping Fundamentals (3 minutes) | プロトタイピングの基礎（3分）

**English Speaking Points:**
- "While wireframes show structure, prototypes show behavior. Prototypes are interactive - users can click, tap, and navigate."
- "A prototype is like a working model of your app. It's not the real thing, but it looks and feels close enough to test with users."
- "Prototypes help you answer critical questions: Can users find what they need? Do they understand how to complete tasks? Does the flow make sense?"
- "The beauty of prototypes is you can test and fix problems before any code is written. This saves enormous time and money."

**Japanese Speaking Points:**
- "ワイヤーフレームが構造を示す一方、プロトタイプは動作を示します。プロトタイプはインタラクティブです - ユーザーはクリック、タップ、ナビゲートできます。"
- "プロトタイプはアプリの作業モデルのようなものです。本物ではありませんが、ユーザーでテストするのに十分近い見た目と感触です。"
- "プロトタイプは重要な質問に答えるのに役立ちます：ユーザーは必要なものを見つけられますか？タスクを完了する方法を理解していますか？フローは理にかなっていますか？"
- "プロトタイプの美しさは、コードが書かれる前にテストして問題を修正できることです。これにより、膨大な時間とお金を節約できます。"

**Demonstration:**
- Show interactive prototype example
- Click through navigation flows
- Point out transitions and interactions
- Explain how it simulates final product

**Key Benefits to Highlight:**
- Early problem detection
- Stakeholder communication
- User testing capability
- Cost savings before development

### Slide 4: Wireframes vs. Prototypes | ワイヤーフレームvsプロトタイプ

#### Understanding the Differences (2 minutes) | 違いの理解（2分）

**English Speaking Points:**
- "Let's clarify the difference. Wireframes are static - they show layout. Prototypes are interactive - they show user flows."
- "Wireframes are fast to create. You can sketch multiple layout options in minutes. Prototypes take longer because you're adding interactivity."
- "Think of it as stages: wireframes come first to explore structure, then prototypes add behavior and navigation."
- "Today we focus on wireframes. Next week, we'll turn your wireframes into interactive prototypes."

**Japanese Speaking Points:**
- "違いを明確にしましょう。ワイヤーフレームは静的です - レイアウトを示します。プロトタイプはインタラクティブです - ユーザーフローを示します。"
- "ワイヤーフレームは素早く作成できます。数分で複数のレイアウトオプションをスケッチできます。プロトタイプはインタラクティビティを追加するため、より時間がかかります。"
- "ステージと考えてください：ワイヤーフレームが最初に構造を探索し、次にプロトタイプが動作とナビゲーションを追加します。"
- "今日はワイヤーフレームに焦点を当てます。来週、ワイヤーフレームをインタラクティブプロトタイプに変えます。"

**Visual Comparison:**
- Side-by-side wireframe vs. prototype
- Show progression from simple to complex
- Emphasize when to use each
- Connect to design process stages

### Slide 5: Fidelity Levels Explained | 忠実度レベルの説明

#### Low, Medium, High Fidelity (3 minutes) | 低、中、高忠実度（3分）

**English Speaking Points:**
- "Fidelity refers to how detailed and realistic your design is. Low fidelity is simple and rough. High fidelity looks like the final product."
- "Low-fidelity wireframes use boxes and grayscale. They're perfect for early exploration when you're testing multiple ideas."
- "Medium-fidelity adds actual content and some visual structure. You use real text instead of Lorem ipsum."
- "High-fidelity includes full branding, colors, images, and detailed design. This is what you show stakeholders for approval."
- "For today's activity, we'll work in low to medium fidelity - focus on structure with some real content."

**Japanese Speaking Points:**
- "忠実度はデザインがどれだけ詳細でリアルかを指します。低忠実度はシンプルで粗いです。高忠実度は最終製品のように見えます。"
- "低忠実度ワイヤーフレームは箱とグレースケールを使用します。複数のアイデアをテストする初期探索に最適です。"
- "中忠実度は実際のコンテンツといくつかの視覚的構造を追加します。Lorem ipsumの代わりに実際のテキストを使用します。"
- "高忠実度は完全なブランディング、色、画像、詳細なデザインを含みます。これはステークホルダーの承認のために示すものです。"
- "今日のアクティビティでは、低から中忠実度で作業します - いくつかの実際のコンテンツを持つ構造に焦点を当てます。"

**Examples to Show:**
- Low-fi: Boxes and lines
- Mid-fi: Content hierarchy visible
- High-fi: Polished, production-ready
- Explain progression path

**When to Use Each:**
- Low-fi: Brainstorming, early feedback
- Mid-fi: Team alignment, content validation
- High-fi: Stakeholder approval, developer handoff

### Slide 9: Wireframing Best Practices | ワイヤーフレーミングベストプラクティス

#### Do's and Don'ts (2 minutes) | すべきこととすべきでないこと（2分）

**English Speaking Points:**
- "Let's talk about what makes a good wireframe versus a bad one."
- "DO focus on layout and content hierarchy. Make the most important content largest and highest on the screen."
- "DO use consistent alignment and spacing. Line up elements on a grid - this creates visual order."
- "DON'T add colors, detailed graphics, or brand elements yet. Save those for high-fidelity design."
- "DON'T spend too much time perfecting one wireframe. Create multiple versions quickly and compare."
- "The best wireframes are simple enough to create in 15-20 minutes but clear enough to communicate the layout."

**Japanese Speaking Points:**
- "良いワイヤーフレームと悪いワイヤーフレームの違いについて話しましょう。"
- "レイアウトとコンテンツ階層に焦点を当ててください。最も重要なコンテンツを最大かつ画面の最上部に配置します。"
- "一貫した整列とスペーシングを使用してください。要素をグリッドに並べます - これは視覚的な秩序を作成します。"
- "まだ色、詳細なグラフィック、またはブランド要素を追加しないでください。それらは高忠実度デザインのために保存してください。"
- "1つのワイヤーフレームを完璧にすることに多くの時間を費やさないでください。複数のバージョンを迅速に作成して比較してください。"
- "最高のワイヤーフレームは15-20分で作成できるほどシンプルですが、レイアウトを伝えるのに十分明確です。"

**Common Mistakes:**
- Too much detail too soon
- Inconsistent spacing
- Ignoring mobile constraints
- No content hierarchy
- Forgetting navigation

### Slide 10: Mobile Design Considerations | モバイルデザインの考慮事項

#### Mobile-First Design (2 minutes) | モバイルファーストデザイン（2分）

**English Speaking Points:**
- "Most apps are used on mobile devices, so we design mobile-first."
- "iPhone screens are typically 375 x 812 pixels. Android varies, but 360 x 640 is common. Use these as your starting point."
- "Touch targets are crucial. Buttons should be at least 44 x 44 pixels - large enough for a finger to tap accurately."
- "Think about one-handed use. Most important actions should be within thumb reach at the bottom of the screen."
- "Mobile users scroll vertically, so design for scrolling rather than cramming everything on one screen."

**Japanese Speaking Points:**
- "ほとんどのアプリはモバイルデバイスで使用されるため、モバイルファーストでデザインします。"
- "iPhone画面は通常375 x 812ピクセルです。Androidは異なりますが、360 x 640が一般的です。これらを出発点として使用してください。"
- "タッチターゲットは重要です。ボタンは少なくとも44 x 44ピクセルである必要があります - 指で正確にタップするのに十分な大きさ。"
- "片手使用について考えてください。最も重要なアクションは画面の下部で親指の届く範囲内にある必要があります。"
- "モバイルユーザーは縦にスクロールするため、すべてを1つの画面に詰め込むのではなく、スクロールのためにデザインしてください。"

**Demonstration:**
- Show mobile screen sizes
- Demonstrate thumb reach zones
- Point out touch target sizes
- Show scrolling vs. cramming

### Slide 11: Common Wireframe Patterns | 一般的なワイヤーフレームパターン

#### Mobile App Patterns (2 minutes) | モバイルアプリパターン（2分）

**English Speaking Points:**
- "Let's look at common patterns you'll use in your wireframes."
- "Homepage/Dashboard typically has a top navigation with title and menu, a hero section for main content, cards for grouped information, and bottom tab bar for primary navigation."

- "Detail views show a back button, large image at top, organized content sections, and a call-to-action button."

**Japanese Speaking Points:**
- "ワイヤーフレームで使用する一般的なパターンを見てみましょう。"

- "リストビューには、検索バー、サムネイルとテキストを含む繰り返されるリストアイテム、アクションボタンが含まれます。"
- "詳細ビューには、戻るボタン、上部の大きな画像、整理されたコンテンツセクション、行動喚起ボタンが表示されます。"
- "フォームには、明確なラベル、入力フィールド、ガイダンス用のヘルパーテキスト、目立つ送信ボタンがあります。"

### Slide 14: Common Wireframing Mistakes | 一般的なワイヤーフレーミングの間違い

**English Speaking Points:**

- "Make sure you have clear content hierarchy - not everything can be equally important."

- "スペーシングに注意してください。全体で一貫したマージンとパディングを使用してください。"
- "モバイル制約を無視しないでください。実際のモバイル画面サイズ用にデザインしてください。"
- "明確なコンテンツ階層があることを確認してください - すべてが同等に重要であることはできません。"

## Common Student Issues and Solutions | 一般的な学生の問題と解決策

### Technical Issues | 技術的問題

#### Issue 1: Can't Access Design Tool
**Symptoms:** Login problems, tool won't load, account creation fails
**Solutions:**
- Verify student email address
- Try different web browser
- Clear browser cache
- Use desktop app instead of web
- Check internet connectivity
- Contact tool support if persistent

#### Issue 2: Tool is Slow or Laggy
**Symptoms:** Delayed response, crashes, freezing
**Solutions:**
- Close other browser tabs/applications
- Reduce zoom level
- Simplify complex designs
- Use desktop app if on web
- Check computer RAM availability
- Save and restart tool

#### Issue 3: Can't Find Tools/Features
**Symptoms:** Lost in interface, can't locate specific functions
**Solutions:**
- Use bot for tool navigation help
- Watch quick tutorial video
- Use tool's built-in search
- Refer to keyboard shortcuts
- Ask peer for guidance
- Follow along with instructor demo

### Design Issues | デザイン問題

#### Issue 1: Too Much Detail Too Soon
**Symptoms:** Adding colors, images, detailed graphics
**Solutions:**
- Remind about fidelity levels
- Show low-fi examples
- Encourage grayscale only
- Focus on structure first
- Explain benefits of simplicity

#### Issue 2: Unclear Layout
**Symptoms:** Confusing hierarchy, poor organization
**Solutions:**
- Review content hierarchy principles
- Look at example wireframes
- Use grid system for alignment
- Simplify and remove clutter
- Get bot feedback on clarity

#### Issue 3: Not Mobile-Appropriate
**Symptoms:** Desktop-like layouts, small touch targets
**Solutions:**
- Check screen dimensions
- Review mobile patterns
- Demonstrate thumb reach zones
- Show mobile app examples
- Adjust touch target sizes

#### Issue 4: Missing Navigation
**Symptoms:** No way to move between screens
**Solutions:**
- Review navigation elements
- Show common patterns (tabs, back buttons)
- Explain user flow importance
- Add navigation to each screen
- Plan multi-screen journey

### Process Issues | プロセスの問題

#### Issue 1: Analysis Paralysis
**Symptoms:** Can't start, overthinking, perfectionism
**Solutions:**
- Set time limits (5 min per screen)
- Start with paper sketch
- Use template as starting point
- Encourage iteration mentality
- Remind that changes are easy

#### Issue 2: Working Too Slowly
**Symptoms:** Spending too much time on details
**Solutions:**
- Set mini-deadlines
- Focus on completion over perfection
- Skip placeholder text for now
- Use copy/paste for repeated elements
- Encourage speed over polish

#### Issue 3: Not Using Resources
**Symptoms:** Struggling alone, not asking for help
**Solutions:**
- Actively offer assistance
- Encourage bot usage
- Promote peer collaboration
- Demonstrate asking for help
- Create safe learning environment
